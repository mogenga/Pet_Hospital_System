# 宠物医院诊疗与住院管理系统——数据库设计文档

## 一、系统概述

### 1.1 项目背景

宠物医院日常业务涵盖门诊接诊、诊断处方、住院护理、药品进销存、宠物寄养以及收费结算等环节。传统纸质管理方式存在信息分散、数据难以复用、统计效率低等问题。本系统通过关系型数据库对上述业务进行统一建模，实现数据集中管理与跨模块联动。

### 1.2 系统功能模块

| 模块 | 核心功能 |
|------|----------|
| 基础档案 | 客户信息维护、宠物档案管理 |
| 诊疗管理 | 挂号就诊、诊断记录、处方开具 |
| 住院护理 | 住院登记、病房/笼位分配、护理日志 |
| 药品库存 | 药品档案维护、批次入库、库存实时更新 |
| 寄养管理 | 寄养登记、笼位占用、费用核算 |
| 收费结算 | 账单生成、费用明细记录、结算状态跟踪 |

### 1.3 开发环境

- 数据库：PostgreSQL 14+
- 字符集：UTF8

## 二、数据库设计说明

### 2.1 设计原则

**范式要求**：全部 15 张表均满足第三范式（3NF），即满足原子性（1NF）、消除部分函数依赖（2NF）、消除传递函数依赖（3NF）。

**主键策略**：统一采用 `SERIAL` 类型自增代理主键，与业务无耦合，便于维护。

**外键约束**：所有关联关系均通过 `FOREIGN KEY` 显式声明，保证参照完整性。

**Check 约束**：对金额、数量等数值字段添加范围约束，在数据库层拦截非法数据。

**注释方式**：PostgreSQL 不支持行内 `COMMENT` 子句，统一使用 `COMMENT ON TABLE / COLUMN` 语句单独添加注释。

### 2.2 MySQL → PostgreSQL 主要语法差异

| 项目 | MySQL | PostgreSQL |
|------|-------|------------|
| 自增主键 | `INT AUTO_INCREMENT` | `SERIAL` |
| 小数类型 | `DECIMAL(m,n)` | `NUMERIC(m,n)` |
| 当前时间默认值 | `DEFAULT CURRENT_TIMESTAMP` | `DEFAULT NOW()` |
| 日期差计算 | `DATEDIFF(d1, d2)` → 返回天数 | `d1 - d2` → 直接返回天数（INTEGER） |
| 当前日期 | `CURDATE()` | `CURRENT_DATE` |
| 字段注释 | `COMMENT '...'`（行内） | `COMMENT ON COLUMN ... IS '...'`（独立语句） |
| 触发器 | 函数体直接写在 `BEGIN...END` 内，`DELIMITER` 分隔 | 先定义 `FUNCTION ... RETURNS TRIGGER`，再 `CREATE TRIGGER ... EXECUTE FUNCTION` |
| 分隔符 | 需要 `DELIMITER $$` 切换 | 使用 `$$` 作为函数体引号，无需切换分隔符 |

### 2.3 3NF 核心调整说明

**问题一：boarding.daily_rate 的传递依赖**

原设计存在如下传递依赖链：

```
boarding_id → ward_id → daily_rate
```

修正方案：将 `daily_rate` 迁移至 `ward` 表，寄养费用通过以下方式动态计算：

```sql
w.daily_rate * (COALESCE(b.end_date, CURRENT_DATE) - b.start_date)
```

**问题二：bill.total_amount 的派生属性**

账单总金额可由 `SUM(bill_item.amount)` 得出，冗余存储会引发不一致风险。修正方案：删除该字段，创建视图 `v_bill_total` 动态聚合，保证数据永远与明细一致。

## 三、表结构详细设计

### 3.1 customer（客户表）

存储宠物主人的基本信息，作为宠物档案的归属主体。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| customer_id | SERIAL | PK | 客户编号（自增） |
| name | VARCHAR(50) | NOT NULL | 客户姓名 |
| phone | VARCHAR(20) | NOT NULL, UNIQUE | 联系电话 |
| address | VARCHAR(200) | — | 家庭地址 |

### 3.2 pet（宠物表）

存储宠物基本档案，通过外键关联所属客户，支持一位客户拥有多只宠物。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| pet_id | SERIAL | PK | 宠物编号（自增） |
| customer_id | INT | NOT NULL, FK → customer | 所属客户 |
| name | VARCHAR(50) | NOT NULL | 宠物名称 |
| species | VARCHAR(30) | NOT NULL | 物种（犬/猫/兔等） |
| breed | VARCHAR(50) | — | 品种 |
| birth_date | DATE | — | 出生日期 |

### 3.3 employee（员工表）

记录医院全体员工信息，通过 `role` 字段区分岗位，供就诊接诊与护理操作关联使用。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| employee_id | SERIAL | PK | 员工编号（自增） |
| name | VARCHAR(50) | NOT NULL | 员工姓名 |
| role | VARCHAR(20) | NOT NULL | 角色：医生/护士/管理员 |
| phone | VARCHAR(20) | NOT NULL, UNIQUE | 联系电话 |

### 3.4 account（账号表）

存储员工登录认证信息，与 `employee` 一对一关联。将认证字段从业务档案中分离，实现职责分离——`employee` 是业务档案（姓名、岗位、联系方式），`account` 是系统凭证（用户名、密码、登录状态），两者更新频率和关注点完全不同。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| account_id | SERIAL | PK | 账号编号（自增） |
| employee_id | INT | NOT NULL, FK → employee, UNIQUE | 关联员工（一对一） |
| username | VARCHAR(50) | NOT NULL, UNIQUE | 登录用户名 |
| password_hash | VARCHAR(255) | NOT NULL | bcrypt 哈希值，禁止存明文 |
| is_active | BOOLEAN | NOT NULL, DEFAULT TRUE | 账号是否启用，禁用后无法登录 |
| last_login | TIMESTAMP | — | 最近登录时间 |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | 创建时间 |

**登录流程**：前端提交 username + password → 查 `account` 表取 `password_hash` → bcrypt 校验 → 校验通过后联查 `employee.role` → 签发 JWT（payload 带 `account_id` 和 `role`）→ 后续请求凭 `role` 做接口级权限控制。

**3NF 说明**：`account` 所有字段均直接依赖 `account_id`，不引入传递依赖，不影响原有 14 张表的 3NF 合规性。

### 3.5 visit（就诊记录表）

每次挂号产生一条就诊记录，关联接诊医生与就诊宠物，是整个诊疗流程的起点。住院状态完全解耦，由 `hospitalization` 表独立管理——判断当前是否住院查询 `hospitalization WHERE visit_id = ? AND status = '住院中'`。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| visit_id | SERIAL | PK | 就诊编号（自增） |
| pet_id | INT | NOT NULL, FK → pet | 就诊宠物 |
| employee_id | INT | NOT NULL, FK → employee | 接诊医生 |
| visit_time | TIMESTAMP | NOT NULL, DEFAULT NOW() | 就诊时间 |
| complaint | VARCHAR(500) | — | 主诉 |
| status | VARCHAR(20) | NOT NULL, DEFAULT '待接诊', CHECK | 待接诊/接诊中/待收费/已完成/已取消 |

**状态流转：**

```
待接诊 ──(accept)──→ 接诊中 ──(diagnosis)──→ 待收费 ──(settle)──→ 已完成
   │                    │
   └──(cancel)──→ 已取消  └──(cancel, 无诊断)──→ 已取消
```

**取消规则：** 状态为待接诊可直接取消；状态为接诊中且无诊断记录可取消；已有诊断后不可取消。

### 3.6 diagnosis（诊断记录表）

存储医生对本次就诊的诊断结论与医嘱。一次就诊对应至多一条诊断记录（`visit_id` 唯一约束）。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| diagnosis_id | SERIAL | PK | 诊断编号（自增） |
| visit_id | INT | NOT NULL, FK → visit, UNIQUE | 关联就诊（一对一） |
| diagnosis_result | VARCHAR(200) | NOT NULL | 诊断结论 |
| notes | TEXT | — | 医嘱备注 |

### 3.7 prescription_item（处方明细表）

记录每个诊断下开具的每一种药品，直接关联药品批次（而非药品档案），便于精确追溯药品来源批次及效期。库存扣减由后端 service 层使用 `SELECT ... FOR UPDATE` 行级锁控制。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| item_id | SERIAL | PK | 明细编号（自增） |
| diagnosis_id | INT | NOT NULL, FK → diagnosis | 所属诊断 |
| batch_id | INT | NOT NULL, FK → medicine_batch | 药品批次 |
| quantity | INT | NOT NULL, CHECK > 0 | 数量 |
| dosage | VARCHAR(100) | — | 用法用量 |

### 3.8 ward（病房/笼位表）

统一管理住院病房与寄养笼位，通过 `type` 字段区分。`daily_rate` 是笼位固有属性，在此统一维护以满足 3NF。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| ward_id | SERIAL | PK | 笼位编号（自增） |
| ward_no | VARCHAR(20) | NOT NULL, UNIQUE | 笼位编号（如 A01） |
| type | VARCHAR(20) | NOT NULL | 普通病房/ICU/寄养笼 |
| status | VARCHAR(20) | NOT NULL, DEFAULT '空闲' | 空闲/占用 |
| daily_rate | NUMERIC(8,2) | NOT NULL, CHECK ≥ 0 | 每日费率（元） |

### 3.9 hospitalization（住院记录表）

记录宠物的住院信息，关联来源就诊与分配笼位。住院生命周期独立管理，与门诊状态（`visit.status`）完全解耦。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| hosp_id | SERIAL | PK | 住院编号（自增） |
| visit_id | INT | NOT NULL, FK → visit, UNIQUE | 来源就诊（一对一） |
| ward_id | INT | NOT NULL, FK → ward | 所在笼位 |
| admit_date | DATE | NOT NULL | 入院日期 |
| discharge_date | DATE | — | 出院日期 |
| status | VARCHAR(20) | NOT NULL, DEFAULT '住院中', CHECK | 住院中/已出院 |

### 3.10 nursing_record（护理记录表）

记录住院期间每次护理操作的时间、执行人员与具体内容，支持一次住院产生多条护理日志。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| record_id | SERIAL | PK | 记录编号（自增） |
| hosp_id | INT | NOT NULL, FK → hospitalization | 所属住院 |
| employee_id | INT | NOT NULL, FK → employee | 执行护士 |
| record_time | TIMESTAMP | NOT NULL, DEFAULT NOW() | 护理时间 |
| content | TEXT | NOT NULL | 护理内容 |

### 3.11 medicine（药品档案表）

记录药品基础信息，不涉及库存与批次，与 `medicine_batch` 分离体现单一职责。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| medicine_id | SERIAL | PK | 药品编号（自增） |
| name | VARCHAR(100) | NOT NULL | 药品名称 |
| unit | VARCHAR(20) | NOT NULL | 单位（片/支/ml） |
| unit_price | NUMERIC(8,2) | NOT NULL, CHECK ≥ 0 | 零售单价（元） |
| category | VARCHAR(50) | NOT NULL | 分类（抗生素/消炎/疫苗等） |

### 3.12 medicine_batch（药品批次表）

每次入库产生一条批次记录，是进销存管理的核心表。`stock_qty` 由后端 service 层使用 `SELECT ... FOR UPDATE` 行级锁控制扣减，`CHECK (stock_qty >= 0)` 在数据库层作为最后防线防止超量出库。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| batch_id | SERIAL | PK | 批次编号（自增） |
| medicine_id | INT | NOT NULL, FK → medicine | 所属药品 |
| in_date | DATE | NOT NULL | 入库日期 |
| expire_date | DATE | NOT NULL, CHECK > in_date | 有效期至 |
| stock_qty | INT | NOT NULL, DEFAULT 0, CHECK ≥ 0 | 当前库存量 |
| cost_price | NUMERIC(8,2) | NOT NULL, CHECK ≥ 0 | 批次进货单价（元） |

### 3.13 boarding（寄养记录表）

记录宠物寄养的起止时间与使用笼位。`daily_rate` 已移至 `ward` 表，寄养费用通过 `ward.daily_rate × (end_date - start_date)` 动态计算。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| boarding_id | SERIAL | PK | 寄养编号（自增） |
| pet_id | INT | NOT NULL, FK → pet | 寄养宠物 |
| ward_id | INT | NOT NULL, FK → ward | 使用笼位 |
| start_date | DATE | NOT NULL | 开始日期 |
| end_date | DATE | — | 结束日期（NULL = 寄养中） |

### 3.14 bill（账单主表）

每次就诊对应一张账单（`visit_id` 唯一约束），仅记录状态与开单时间，总金额通过视图 `v_bill_total` 动态汇总。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| bill_id | SERIAL | PK | 账单编号（自增） |
| visit_id | INT | NOT NULL, FK → visit, UNIQUE | 关联就诊（一对一） |
| status | VARCHAR(20) | NOT NULL, DEFAULT '未结清' | 未结清/已结清 |
| created_at | TIMESTAMP | NOT NULL, DEFAULT NOW() | 开单时间 |

### 3.15 bill_item（账单明细表）

记录账单下的每一笔收费，通过 `item_type` 区分费用类别，支持按类别统计分析。`source_type` + `source_id` 组合唯一约束保证幂等——同一来源的同一笔收费不会因前端重试而重复插入。

| 字段名 | 数据类型 | 约束 | 说明 |
|--------|----------|------|------|
| bill_item_id | SERIAL | PK | 明细编号（自增） |
| bill_id | INT | NOT NULL, FK → bill | 所属账单 |
| item_type | VARCHAR(30) | NOT NULL | 诊疗费/药品费/住院费/寄养费 |
| source_type | VARCHAR(20) | NOT NULL | 来源类型：diagnosis/prescription/hospitalization/boarding |
| source_id | INT | NOT NULL | 来源记录主键 |
| description | VARCHAR(200) | — | 费用说明 |
| amount | NUMERIC(8,2) | NOT NULL, CHECK ≥ 0 | 金额（元） |

**唯一约束：** `UNIQUE (bill_id, source_type, source_id)` —— 同一账单下，同一来源的同一记录只能出一次费用。前端重试 POST → 数据库冲突 → service 层捕获后返回 200（幂等），不重复插入。

## 四、表间关系说明

### 4.1 主要关系汇总

| 关系 | 类型 | 说明 |
|------|------|------|
| customer → pet | 1 : N | 一位客户可拥有多只宠物 |
| pet → visit | 1 : N | 一只宠物可有多次就诊记录 |
| employee → account | 1 : 0..1 | 一位员工至多一个登录账号（唯一约束） |
| employee → visit | 1 : N | 一位医生可接诊多次 |
| visit → diagnosis | 1 : 0..1 | 一次就诊至多一条诊断（唯一约束） |
| diagnosis → prescription_item | 1 : N | 一条诊断可开具多种药品 |
| medicine_batch → prescription_item | 1 : N | 处方明细关联具体批次 |
| medicine → medicine_batch | 1 : N | 同一药品可有多个入库批次 |
| visit → hospitalization | 1 : 0..1 | 就诊后可转住院（唯一约束） |
| ward → hospitalization | 1 : N | 一个笼位可先后安置不同宠物 |
| hospitalization → nursing_record | 1 : N | 一次住院可有多条护理记录 |
| employee → nursing_record | 1 : N | 一位护士可执行多次护理 |
| pet → boarding | 1 : N | 一只宠物可多次寄养 |
| ward → boarding | 1 : N | 一个笼位可先后接受不同寄养 |
| visit → bill | 1 : 0..1 | 一次就诊对应一张账单（唯一约束） |

### 4.2 核心业务流程数据流向

```
customer ──拥有──> pet ──就诊──> visit ──产生──> diagnosis ──开具──> prescription_item
                                   │                                         │
                                   ├──转住院──> hospitalization               └──关联── medicine_batch
                                   │               │                                        │
                                   │               └──记录──> nursing_record      medicine ─┘
                                   │
                                   ├──生成──> bill ──包含──> bill_item
                                   │
                                   └──(宠物) ──寄养──> boarding ──使用──> ward
```

## 五、视图设计

### 5.1 v_bill_total（账单总金额视图）

**用途**：替代原 `bill.total_amount` 字段，动态聚合账单明细，确保总金额与明细数据强一致。

```sql
CREATE OR REPLACE VIEW v_bill_total AS
SELECT
    b.bill_id,
    b.visit_id,
    b.status,
    b.created_at,
    COALESCE(SUM(i.amount), 0) AS total_amount
FROM bill b
LEFT JOIN bill_item i ON b.bill_id = i.bill_id
GROUP BY b.bill_id, b.visit_id, b.status, b.created_at;
```

**使用示例**：

```sql
-- 查询指定账单的总金额
SELECT * FROM v_bill_total WHERE bill_id = 1;

-- 查询所有未结清账单及金额
SELECT * FROM v_bill_total WHERE status = '未结清';
```

## 六、库存控制策略

> 原 `trg_reduce_stock` 触发器已移除。业务逻辑不应在数据库层实现，改为后端 service 层控制。

### 6.1 库存扣减（后端 service 层）

处方开具时，后端使用 `SELECT ... FOR UPDATE` 行级锁扣减库存：

```python
async def deduct_stock(db: AsyncSession, batch_id: int, qty: int):
    batch = await db.get(MedicineBatch, batch_id, with_for_update=True)
    if batch.stock_qty < qty:
        medicine = await db.get(Medicine, batch.medicine_id)
        raise Conflict(f"药品「{medicine.name}」库存不足：需要 {qty}，当前库存 {batch.stock_qty}")
    batch.stock_qty -= qty
```

### 6.2 数据库层防线

`medicine_batch.stock_qty` 保留 `CHECK (stock_qty >= 0)` 约束。若后端 bug 导致超卖穿透 service 层，数据库仍会拒绝负数写入并回滚事务，提供最后一道安全屏障。

## 七、典型业务查询

### 7.1 查询某宠物全部就诊历史

```sql
SELECT
    v.visit_id,
    v.visit_time,
    v.complaint,
    e.name             AS doctor,
    d.diagnosis_result,
    d.notes
FROM visit v
JOIN employee  e ON v.employee_id = e.employee_id
LEFT JOIN diagnosis d ON v.visit_id = d.visit_id
WHERE v.pet_id = 1
ORDER BY v.visit_time DESC;
```

### 7.2 查询某次就诊的处方详情

```sql
SELECT
    m.name                        AS medicine_name,
    m.unit,
    mb.expire_date,
    pi.quantity,
    pi.dosage,
    m.unit_price * pi.quantity    AS subtotal
FROM prescription_item pi
JOIN medicine_batch mb ON pi.batch_id    = mb.batch_id
JOIN medicine       m  ON mb.medicine_id = m.medicine_id
WHERE pi.diagnosis_id = 1;
```

### 7.3 查询账单总金额（使用视图）

```sql
SELECT
    bt.bill_id,
    p.name        AS pet_name,
    bt.created_at,
    bt.status,
    bt.total_amount
FROM v_bill_total bt
JOIN visit v ON bt.visit_id = v.visit_id
JOIN pet   p ON v.pet_id    = p.pet_id
WHERE bt.bill_id = 1;
```

### 7.4 计算寄养费用

PostgreSQL 中两个 `DATE` 类型直接相减即得天数（`INTEGER`），无需 `DATEDIFF`。

```sql
SELECT
    b.boarding_id,
    p.name                                                                  AS pet_name,
    w.ward_no,
    w.daily_rate,
    (COALESCE(b.end_date, CURRENT_DATE) - b.start_date)                    AS days,
    w.daily_rate * (COALESCE(b.end_date, CURRENT_DATE) - b.start_date)     AS total_fee
FROM boarding b
JOIN pet  p ON b.pet_id  = p.pet_id
JOIN ward w ON b.ward_id = w.ward_id;
```

### 7.5 查询库存不足的药品批次（库存 < 10）

```sql
SELECT
    m.name        AS medicine_name,
    mb.batch_id,
    mb.expire_date,
    mb.stock_qty
FROM medicine_batch mb
JOIN medicine m ON mb.medicine_id = m.medicine_id
WHERE mb.stock_qty < 10
ORDER BY mb.stock_qty ASC;
```

### 7.6 按费用类型统计某月收入

```sql
SELECT
    bi.item_type,
    SUM(bi.amount) AS total_revenue
FROM bill_item bi
JOIN bill b ON bi.bill_id = b.bill_id
WHERE b.created_at BETWEEN '2025-01-01' AND '2025-01-31'
GROUP BY bi.item_type
ORDER BY total_revenue DESC;
```

## 八、数据库完整性约束汇总

| 约束类型 | 涉及字段 | 说明 |
|----------|----------|------|
| 主键约束 | 全部 `*_id` 字段 | SERIAL 自增，唯一标识每条记录 |
| 唯一约束 | customer.phone | 客户电话不可重复 |
| 唯一约束 | employee.phone | 员工电话不可重复 |
