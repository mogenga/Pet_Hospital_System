## 一、visit.status 语义混乱

### 问题

原数据库设计中 `visit.status` 仅包含三个值：`进行中 / 已完成 / 已转住院`。这在业务流程中存在两个致命缺陷：

**缺陷一：状态粒度太粗。**

"已完成"在业务上承载了三件事——诊疗结束、不再开药、账单结清。但实际流程中这三件事是分离的：医生写完诊断不代表管理员收了钱，更不代表患者可以离院。把三个不同时间点发生的事塞进一个"已完成"，导致任何人都无法从 visit.status 判断这张就诊单到底处于哪个阶段。

举个例子：医生下了诊断、开了药，visit.status 变成"已完成"。但药房还没发药，收费处还没结账。此时护士在系统里看到"已完成"，可能会认为"结束了"而忽略后续流程。

**缺陷二：两个业务流揉进一个字段。**

"已转住院"把住院作为 visit 状态的终态之一。问题是——住院不是 visit 的终态，而是 visit 产生的一个**结果**。一个 visit 转住院后，门诊这边还是需要收费（住院费和门诊费可能是同一张账单），visit 本身的生命周期并没有因为转住院就结束。

而且"已转住院"和"已完成"之间存在竞争关系：转了住院 = 门诊结束了吗？如果住院期间需要补开门诊处方怎么办？语义完全混乱。

### 讨论

一开始讨论的方案是在 visit 上加一个 `is_hospitalized BOOLEAN` 字段，让 status 保留原有值但额外标记住院状态。但这样就形成了两个字段描述同一件事（visit 的住院情况），数据冗余且可能出现 `status='已完成' AND is_hospitalized=TRUE` 这种矛盾组合。

最终决定：**拆分 visit.status 为五个精确的阶段状态，同时完全解耦住院。** 住院有自己的 `hospitalization.status`（住院中/已出院），两个生命周期互不干扰。

### 解决方案

visit 的状态机重新设计为门诊专属流程：

```
待接诊 ──(accept)──→ 接诊中 ──(diagnosis)──→ 待收费 ──(settle)──→ 已完成
   │                    │
   └──(cancel)──→ 已取消  └──(cancel, 无诊断)──→ 已取消
```

五个状态的语义：

| 状态 | 含义 | 允许的操作 |
|------|------|-----------|
| 待接诊 | 管理员挂号完成，等待医生接诊 | 接诊、取消 |
| 接诊中 | 医生正在处理 | 诊断（含开处方）、取消（仅无诊断时） |
| 待收费 | 诊断完成，等待结账 | 生成收费项、结账、转住院 |
| 已完成 | 结账完毕，关单 | 无（终态） |
| 已取消 | 管理员取消 | 无（终态） |

住院通过 `hospitalization` 表独立追踪：`hospitalization.status IN ('住院中','已出院')`。判断 visit 是否正在住院通过 `EXISTS (SELECT 1 FROM hospitalization WHERE visit_id=? AND status='住院中')`。

### 影响范围

- `pet_hospital_pg.sql`：visit 表 DDL（status CHECK + DEFAULT 变更）
- `docs/数据库设计文档.md`：visit 表结构 + 状态流转说明
- `consultation/service.py`：所有 status 判断逻辑
- 前端：visit 列表的状态筛选、详情页状态标签

## 二、is_hospitalized 冗余字段

### 问题

在问题一的早期讨论中，曾考虑在 visit 表增加 `is_hospitalized BOOLEAN` 字段作为过渡方案。但经过深入分析，该字段存在两个根本性问题：

**问题一：名称天然歧义。** `is_hospitalized = TRUE` 可以有两种解读：
- "当前正在住院"（需要出院时设回 FALSE）
- "曾经住过院"（历史标记，永远不设回 FALSE）

两种解读对应完全不同的数据维护策略。一个人写成"当前住院"，另一个人当作"历史标记"，上线后必然出现数据一致性问题。

**问题二：违反单一真相源原则。** `hospitalization` 表已经通过 `UNIQUE(visit_id)` 保证了一个 visit 最多一次住院。visit 是否转住院完全可以从 hospitalization 表推导出来。冗余一个布尔字段，两者可能不同步——比如 hospitalization 已出院但 `is_hospitalized` 没回设，或者反过来。

### 讨论

两种方案对比：

| | A. is_hospitalized 布尔 | B. 完全依赖 hospitalization EXISTS |
|---|---|---|
| 数据一致性 | 需维护两处，可能不同步 | 单一真相源 |
| 语义 | 模糊（当前？曾经？） | 精确（SQL 自行定义条件） |
| 查询成本 | 直接读字段 | 子查询 EXISTS，但有索引几乎无差别 |
| 扩展性 | 只能标记是否，无法区分详情 | 可扩展（加 status/时间等） |

### 解决方案

**删除 `is_hospitalized` 字段，不新增任何冗余存储。** 住院状态完全由 `hospitalization` 表权威维护。

同时 `hospitalization` 新增 `status VARCHAR(20) CHECK ('住院中','已出院')`，令住院生命周期可以独立管理而非仅靠 `discharge_date IS NULL` 隐式表达。

判断函数：
- 是否当前在院：`EXISTS (SELECT 1 FROM hospitalization WHERE visit_id=? AND status='住院中')`
- 是否曾住院：`EXISTS (SELECT 1 FROM hospitalization WHERE visit_id=?)`

### 影响范围

- `pet_hospital_pg.sql`：visit 表不新增 is_hospitalized，hospitalization 新增 status 字段 + CHECK
- 所有需要判断住院状态的查询改为 EXISTS 子查询

## 三、数据库触发器存放业务逻辑

### 问题

原设计在 PostgreSQL 中定义了触发器 `trg_reduce_stock`，每当向 `prescription_item` 插入数据时，自动扣减 `medicine_batch.stock_qty`。这个做法在架构审查中被否决。

业务逻辑放在数据库层的三个固有问题：

**不可测试。** 触发器是数据库对象的副作用，无法在应用层的单元测试中被覆盖。测试处方开具流程时，库存扣减是一个隐式的、不可 mock 的操作，导致要么依赖真实数据库（慢），要么完全跳过测试（危险）。

**不可调试。** 当库存出现异常（比如被扣成负数）时，开发者需要追溯"是哪个操作触发的扣减"。触发器的调用栈隐藏在数据库内核执行过程中，应用日志里没有任何记录。而且多个表可能都有触发器链式触发，排查路径非常曲折。

**依赖 DB 专有特性。** PostgreSQL 的触发器函数用 PL/pgSQL 编写，语法与 Python 完全不同。如果未来需要迁移到其他数据库（MySQL、TiDB），触发器是迁移成本最高的部分。

### 讨论

保留 CHECK 约束作为最后防线，但库存扣减逻辑移到后端。这样做的代价是：应用程序多了几行代码，且需要显式使用行级锁防止并发问题。收益是：逻辑可测试、可调试、可移植。

### 解决方案

1. 删除 `fn_reduce_stock()` 函数和 `trg_reduce_stock` 触发器
2. 库存扣减移至后端 `pharmacy/service.py` 的 `deduct_stock` 函数
3. 使用 SQLAlchemy 的 `with_for_update()` 获取行级锁，防止并发超卖
4. `CHECK (stock_qty >= 0)` 约束保留在数据库中——如果后端 bug 导致超卖穿透 service 层，数据库仍会拒绝负数写入并回滚事务

### 影响范围

- `pet_hospital_pg.sql`：删除触发器区块，替换为库存扣减策略注释
- `docs/数据库设计文档.md`：第六节从"触发器设计"改为"库存控制策略"
- `pharmacy/service.py`：新增 `deduct_stock` 函数
- `consultation/service.py`：处方开具时调用 `deduct_stock`

## 四、bill_item 缺少幂等控制

### 问题

收费接口 `POST /billing/visits/{visit_id}/items` 的设计存在幂等性漏洞。

假设医生在"生成收费项"页面点击"确认"，浏览器发起 POST 请求。由于网络抖动，前端在 3 秒内未收到响应，认为请求失败，自动重试（或用户手动刷新页面再点一次）。后端实际上两次请求都成功处理了——于是 `bill_item` 表里出现了两条完全相同的收费记录。

对于收费系统，重复计费是不可接受的。而依赖前端防抖、按钮 loading 或者"用户不会这么操作"的假设来保证数据正确性，迟早会出问题。

### 讨论

实现幂等有三种常见方案：

| | A. 前端防重 | B. Token 机制 | C. 数据库唯一约束 |
|---|---|---|---|
| 原理 | 按钮 loading / 防抖 | 请求先拿 token，一次性消费 | DB 层直接拒绝重复 |
| 防御范围 | 同页面 | 同页面 + 同 token 内 | 所有场景 |
| 复杂度 | 低 | 中（需 Redis 存 token） | 低 |

C 方案最简单、最可靠。数据库层的唯一约束是对所有路径（前端重试、后端重试、脚本批量导入）的终极防御。

### 解决方案

`bill_item` 增两个字段：

- `source_type VARCHAR(20)`：来源类型（`diagnosis` / `prescription` / `hospitalization` / `boarding`）
- `source_id INT`：来源记录的主键

加唯一约束：`UNIQUE(bill_id, source_type, source_id)`

效果：同一账单下，同一来源的同一笔记录只能插入一次。前端重试 POST → 数据库抛 `IntegrityError`（唯一冲突）→ service 层捕获后按幂等返回已有记录（HTTP 200），不重复插入，对前端透明。

### 影响范围

- `pet_hospital_pg.sql`：bill_item 表新增两个字段 + 唯一约束
- `docs/数据库设计文档.md`：bill_item 表结构更新
- `billing/service.py`：插入逻辑实现幂等捕获

## 五、收费逻辑双入口（BillingService 分叉）

### 问题

在业务流程设计中，门诊收费和住院出院收费都会创建 `bill_item`——门诊收诊疗费和药品费，住院收住院费。但这两个入口在最初的设计中是独立实现的：

- 门诊收费（consultation 模块）：先 INSERT bill (if not exists)，再 INSERT bill_item
- 住院出院（hospitalization 模块）：同样逻辑，但是不同的代码

两个入口写同样的逻辑 = 必然出现行为不一致。比如：门诊入口做了幂等处理而住院入口漏了；门诊入口的错误信息是中文而住院入口是英文；门诊入口支持某类费用而住院入口不支持。随着时间推移，两个分支的代码会越走越远。

### 讨论

解决方案有争议：是让 billing 模块提供共享服务，还是让 consultation 和 hospitalization 各自管自己的收费？最终选择共享服务，原因：
- "创建 bill → 插入 bill_item" 是一个原子操作，应该有一个权威入口
- BillingService 可以统一实现幂等、校验、日志
- 未来如果有新模块（比如体检、美容）也要收费，不需要再造轮子

### 解决方案

抽取 `shared/services/billing_service.py`，门诊和住院统一调用：

**BillingService 接口：**
- `ensure_bill(visit_id)`：幂等获取或创建 bill（`ON CONFLICT visit_id DO NOTHING`）
- `add_item(visit_id, source_type, source_id, amount, description)`：插入收费项，幂等处理唯一冲突

调用方不直接操作 Bill/BillItem 表，所有收费逻辑走 BillingService。

### 影响范围

- 新增 `shared/services/billing_service.py`
- `consultation/service.py` 的收费步骤改为调 BillingService
- `hospitalization/service.py` 的出院步骤改为调 BillingService
- 前端无需变化（API 接口不变）

## 六、护理日志缺少药品追溯

### 问题

MongoDB `nursing_logs` 集合用于存储每次护理的详细日志。在原设计中，护士执行给药的记录是纯文本：

```json
"medication": [
  { "drug": "阿莫西林", "dose": "0.5ml", "route": "皮下注射" }
]
```

从医疗合规角度看，这种设计有风险。药品名称"阿莫西林"在系统里可能存在多个批次（不同厂家、不同效期）。如果发生医疗纠纷——比如患者主诉使用了过期的阿莫西林——从纯文本中无法确定护士从哪个批次取的药。追溯链路在护理日志处断了。

### 讨论

加 `medicine_id` 和 `batch_id` 的争议在于：这增加了护理录入的复杂度——护士需要在表单里从下拉列表选批次，而不是手写药名。但医疗场景下，可追溯性是刚需。如果是普通酒店的客房服务日志，纯文本没问题；但这是宠物医疗机构，需要对用药安全负责。

### 解决方案

`medication` 数组中的每个元素新增两个字段：

```json
{ "medicine_id": 15, "batch_id": 203, "drug": "阿莫西林", "dose": "0.5ml", "route": "皮下注射" }
```

- `medicine_id` 和 `batch_id` 为可选字段——如果历史数据迁移时没有批次信息，可以留空
- 新录入的护理记录**必须**包含这两个字段（前端表单校验 + 后端 schema 校验）

### 影响范围

- MongoDB `nursing_logs` 集合的 medication 子文档结构
- 护理录入表单（前端）：药品选择改为下拉搜索（按药品名 + 批次筛选），不再自由文本
- 后端 `hospitalization/schemas.py`：校验 medicine_id 和 batch_id

## 七、库存扣减错误信息模糊

### 问题

库存扣减失败时的错误提示是 `"库存不足，当前库存 N"`。这传达了两个信息：失败了 + 当前库存量。但缺少最关键的信息——**什么药品缺货了**。

场景：医生给一只狗的处方包含三种药——抗生素（阿莫西林）、止痛药（美洛昔康）、外用药膏（百多邦）。处方提交后返回 409 "库存不足，当前库存 3"。医生不知道是哪种药缺货、缺了多少，需要逐个去库存列表查询排查。如果系统有数百种药品，这种排查可能持续几分钟。

### 讨论

错误的粒度应该与业务上下文匹配。处方开具是一个批量操作（多条 prescription_item 在一个事务里），扣减失败时不仅要告诉用户"什么药缺了"，还要在批量操作中精确定位到具体哪一条。改进后的错误信息模板应包含药品名称、需求量、当前库存三个维度。

### 解决方案

错误信息格式：`"药品「{药品名称}」库存不足：需要 {需求量}，当前库存 {当前量}"`

后端在扣减前先查 Medicine 表获取名称，失败时组织完整信息。前端收到后直接 toast 展示。

连带改进：处方批量操作中多条 `deduct_stock` 调用依次执行，第一条失败即回滚整个事务。错误信息只报第一条失败的药品（因为后续的在回滚前未被验证）。

### 影响范围

- `pharmacy/service.py` 的 `deduct_stock` 函数：扣减前查询 Medicine.name

## 八、MongoDB nursing_logs 结构过度设计

### 问题

`nursing_logs` 的初版设计使用了 `entries` 数组，把一天内多个时间点的护理操作聚合到一条 MongoDB 文档里，类似"护理日报"的结构。这种设计与 PG `nursing_record` 的一对一映射原则矛盾——PG 里 `nursing_record` 是一条一条的（每次护理 = 一行），MongoDB 却做成了按时间段聚合。

这带来的问题是：数据粒度和对应关系不统一。前端展示时，如果 PG 有 3 条记录、MongoDB 有 1 条（内含 3 个 entries），需要做合并逻辑。如果护士只修改其中一条，面临"更新整个 MongoDB 文档"还是"局部更新 entries 中的一项"的选择。

### 讨论

根本原则：**MongoDB 文档和 PG 记录保持 1:1 对应。** 不要因为 MongoDB 支持嵌套数组就去嵌套。结构简单 > 结构灵活。

### 解决方案

一个 `record_id` 严格对应一个 MongoDB 文档：

```
PG nursing_record.record_id ←→ MongoDB nursing_logs.record_id (1:1)
```

文档内 `content` 字段是结构化的（vital_signs / medication / feeding / elimination / notes），但不嵌套多时段。每个时段各自是一份独立的 MongoDB 文档。

### 影响范围

- MongoDB `nursing_logs` 的 Schema 结构
- 前后端数据转换逻辑（不需要做 PG 记录 ↔ MongoDB entries 的聚合/拆分）

## 九、项目结构缺失项

### 问题

架构设计阶段的项目目录树中，日常开发的 4 个文件里漏了 2 个核心项：

**缺失一：SQLAlchemy ORM 模型文件位置不明确。** 目录结构中每个模块只有 `router.py / service.py / schemas.py`，没有 ORM 模型文件。实际开发中，SQLAlchemy 的 Table 定义需要一个明确的归属位置。

**缺失二：CORS 中间件未提及。** 开发环境前后端分离运行（前端 `localhost:5173`，后端 `localhost:8000`），必然涉及跨域。`main.py` 的职责描述里没有 CORS，实施者可能忘了配置，导致前端调试时被浏览器拦截。

### 讨论

关于 Employee 模型放哪里：Employee 和 Account 是一对一认证体系起点。Employee 被多个模块引用（consultation 的 visit.employee_id、hospitalization 的 nursing_record.employee_id），但从业务归属看，员工信息跟着账号体系走，放在 auth 模块。其他模块通过 SQLAlchemy ForeignKey 引用即可。

### 解决方案

1. 每个模块目录增加 `models.py`（SQLAlchemy Table 定义）
2. `main.py` 在 FastAPI app 创建后立即注册 CORS 中间件（`allow_origins=["http://localhost:5173"]`）
3. Employee 模型放在 `modules/auth/models.py`

### 影响范围

- 后端项目脚手架目录结构
- `main.py` 启动配置

## 十、visit 取消规则未定义

### 问题

`visit.status` CHECK 约束中包含了 `已取消` 状态，但在最初设计中只定义了"有取消这个状态"，没有定义取消的业务规则。

三个待回答的问题：
1. **谁有权限取消？** 医生？管理员？护士？
2. **什么状态下可以取消？** 任何状态？还是只在特定状态？
3. **已有诊断的可否取消？** 如果医生已经下了诊断结论并开了药，还能回退吗？

没有明确规则，实施者必然按自己的理解写代码，导致不同人对"取消"的理解不一致。

### 讨论

关于"已有诊断可否取消"：从业务角度看，诊断一旦进入系统就是医疗记录，具有法律效力。取消 = 逻辑删除（标记状态，不物理删除记录），必须保护已产生的诊断不被随意抹掉。权衡后决定：有诊断就不能取消——这是最保守、最保护数据的策略。

关于"谁可以取消"：医生可以取消自己接诊但尚未诊断的 visit（比如客户放弃就诊），管理员有最高权限可以取消任何符合状态条件的 visit，护士不能取消。

### 解决方案

**可取消条件：** 状态为 `待接诊` 或 `接诊中`（且无诊断记录）

| 状态 | 可取消 | 理由 |
|------|--------|------|
| 待接诊 | ✅ | 挂号后尚未接诊，无任何医疗数据 |
| 接诊中（无诊断） | ✅ | 医生接诊但未下诊断，无医疗记录 |
| 接诊中（已有诊断） | ❌ | 诊断具有法律效力，不可回退 |
| 待收费 | ❌ | 诊断已完成，不可回退 |
| 已完成 | ❌ | 终态 |

**操作权限：** 仅管理员。医生如需取消无诊断的 visit，需要通过管理员执行。

**实现位置：** `consultation/service.py`，取消逻辑需先做 status 范围检查，再做 diagnosis 存在性检查。

### 影响范围

- `consultation/service.py`：`cancel_visit` 函数
- 前端：管理员端取消按钮的显隐逻辑（`status IN ('待接诊') OR (status='接诊中' AND 无诊断)`）

## 十一、后端实现计划审查发现的问题（2026-05-01）

以上十条为架构设计阶段发现的问题。在编写实现计划后，计划审查又发现了以下实现层面的问题。这些问题不改变设计，但影响实现方式。

### 11.1 实现顺序依赖错误

**问题：** 初始实现计划把 consultation 排在阶段 4、pharmacy 排在阶段 7。但 consultation 的处方开具调用 pharmacy 的 `deduct_stock` 函数，形成"阶段 4 依赖阶段 7"的死锁。

**解决方案：** 调整顺序——pharmacy 提到阶段 3（auth 之后、consultation 之前）。

### 11.2 TDD 原则未遵守

**问题：** 初始实现计划把所有测试推迟到阶段 12（最后），与架构设计中"写代码同时写测试"的原则直接矛盾。

**解决方案：** 每个模块增加 `Task X.0: 测试先行`，先编写该模块的关键场景测试（失败），再实现（通过）。

### 11.3 缓存序列化方式不当

**问题：** 初始实现计划的 Cache-Aside 示例代码将 SQLAlchemy ORM 对象直接 `**dict` 构造放在 Redis 中。SQLAlchemy ORM 的 `__init__` 不接受这种方式。

**解决方案：** 缓存值统一用 Pydantic schema 做序列化/反序列化：
- 写缓存：ORM → Pydantic `.model_dump()` → JSON → Redis
- 读缓存：Redis → JSON → Pydantic `.model_validate()` → 返回 Pydantic schema

### 11.4 BillingService 幂等异常捕获范围过大

**问题：** `add_item` 的幂等逻辑用 `except Exception` 捕获所有异常，包括系统信号。应只捕获 `IntegrityError`。

**解决方案：** `except sqlalchemy.exc.IntegrityError` + 幂等返回已有记录。

### 11.5 依赖声明遗漏

**问题：** `pyproject.toml` 的依赖列表中有 `pymongo`（同步 MongoDB 驱动），但 `mongo_db.py` 实际使用的是 `motor.motor_asyncio`（异步驱动）。两者是不同的包。

**解决方案：** `pyproject.toml` 依赖改为 `motor`（motor 内部依赖 pymongo，不需要显式声明两个）。
