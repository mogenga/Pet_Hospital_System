### 1.1 各组件职责

| 组件 | 角色 | 说明 |
|------|------|------|
| **PostgreSQL** | 关系型业务核心，唯一真相源 | 客户、宠物、员工、挂号、账单、库存等所有强关系数据。15 张表满足 3NF |
| **MongoDB** | 医疗文书仓库 | 完整病历文档、护理详细日志。通过 `diagnosis_id`/`record_id` 与 PG 弱关联 |
| **Redis** | 缓存 + 会话 | JWT 黑名单、热点数据缓存、接口限流、临时凭证 |
| **MinIO** | 文件/图片存储 | 宠物照片、医学影像、签署文件、账单 PDF 导出 |

### 1.2 关键边界

- PG 唯一真相源，所有写入以 PG 为准
- MongoDB 只做文书增补，通过 `visit_id`/`diagnosis_id` 弱关联回 PG，**无跨库外键**
- Redis 纯缓存/会话层，挂了不影响核心业务（仅性能下降 + 需重新登录）
- MinIO 存文件，PG 只存 `file_key`，不存二进制
- PG 是完全自洽的——即使 MongoDB 挂了，核心业务流程（挂号、开处方、收费）不受影响，只是病历详情暂时不可见

## 二、PostgreSQL 数据库

> 已有 15 张表的设计文档和 SQL 脚本，以下仅记录架构设计阶段产生的变更。完整表结构参见 `docs/数据库设计文档.md`。

### 2.2 visit 表变更

| 变更项 | 旧值 | 新值 | 原因 |
|--------|------|------|------|
| status 默认值 | `'进行中'` | `'待接诊'` | 语义更精确，对齐业务流程 |
| status CHECK | `进行中/已完成/已转住院` | `待接诊/接诊中/待收费/已完成/已取消` | 拆开门诊状态的完整生命周期 |
| 删除冗余标记 | 曾提议新增 `is_hospitalized BOOLEAN` | 不新增，改为查询 hospitalization 表 | `visit` 不冗余住院状态，由 `hospitalization` 表（UNIQUE visit_id + status CHECK）作为权威数据源 |

**visit.status 状态流转规则：**

```
挂号 → 待接诊
医生开始接诊 → 接诊中
完成诊断/处方 → 待收费
管理员结账 → 已完成
管理员取消 → 已取消
```

**取消规则（在 service 层实现）：**

| 状态 | 可取消 |
|------|--------|
| 待接诊 | ✅ |
| 接诊中（无诊断） | ✅ |
| 接诊中（已有诊断） | ❌ |
| 待收费 | ❌ |
| 已完成 | ❌ |

实现逻辑：

```python
async def cancel_visit(db: AsyncSession, visit_id: int):
    visit = await db.get(Visit, visit_id)
    if visit.status not in ("待接诊", "接诊中"):
        raise Conflict("当前状态不可取消")
    diagnosis = await db.scalar(
        select(Diagnosis).where(Diagnosis.visit_id == visit_id)
    )
    if diagnosis:
        raise Conflict("已有诊断记录，不可取消")
    visit.status = "已取消"
```

### 2.2 hospitalization 表变更

| 变更项 | 旧值 | 新值 | 原因 |
|--------|------|------|------|
| 新增字段 | — | `status VARCHAR(20) NOT NULL DEFAULT '住院中'` | 住院生命周期独立管理 |
| status CHECK | — | `住院中/已出院` | — |

住院与门诊完全解耦。判断当前是否住院：`EXISTS (SELECT 1 FROM hospitalization WHERE visit_id = ? AND status = '住院中')`；判断是否曾住院：`EXISTS (SELECT 1 FROM hospitalization WHERE visit_id = ?)`。visit 表不冗余任何住院相关字段，由 hospitalization 作为权威数据源。

### 2.3 bill_item 表变更

| 变更项 | 旧值 | 新值 | 原因 |
|--------|------|------|------|
| 新增字段 | — | `source_type VARCHAR(20) NOT NULL` | 费用来源类型：`diagnosis/prescription/hospitalization/boarding` |
| 新增字段 | — | `source_id INT NOT NULL` | 来源记录的主键 |
| 新增约束 | — | `UNIQUE (bill_id, source_type, source_id)` | 防止前端重试产生重复收费（幂等控制） |

幂等效果：重复 POST 同一收费项 → 数据库唯一冲突 → service 层捕获后返回 200（幂等），不重复插入。

### 2.4 触发器移除

| 移除项 | 原因 |
|--------|------|
| `fn_reduce_stock()` | 业务逻辑不应在数据库层实现，移至后端 service 层用 `SELECT ... FOR UPDATE` 行级锁控制 |
| `trg_reduce_stock` | 同上 |

库存扣减改为后端事务：

```python
async def deduct_stock(db: AsyncSession, batch_id: int, qty: int):
    batch = await db.get(MedicineBatch, batch_id, with_for_update=True)
    if batch.stock_qty < qty:
        medicine = await db.get(Medicine, batch.medicine_id)
        raise Conflict(
            f"药品「{medicine.name}」库存不足：需要 {qty}，当前库存 {batch.stock_qty}"
        )
    batch.stock_qty -= qty
```

`CHECK (stock_qty >= 0)` 保留作为最后防线，防止 bug 导致超卖穿透。

## 四、Redis 缓存策略

### 4.1 用途与 Key 规范

| 用途 | Key 格式 | TTL | 失效时机 |
|------|----------|-----|----------|
| 药品目录缓存 | `medicine:list` | 30 min | 药品入库/编辑后删除 |
| 笼位状态缓存 | `ward:status` | 5 min | 分配/释放笼位后删除 |
| 员工列表缓存 | `employee:list` | 60 min | 编辑员工后删除 |
| 客户列表缓存 | `customer:list` | 10 min | 增改客户后删除 |
| 登录限流 | `ratelimit:login:<ip>` | 15 min | 自然过期（滑动窗口计数） |
| JWT 黑名单 | `jwt:blacklist:<account_id>` | = JWT 剩余有效期 | 修改密码/禁用账号时加入 |
| 密码重置 token | `pwd:reset:<token>` | 30 min | 用完即删 |

### 4.2 缓存模式：Cache-Aside

- **读**：先查 Redis → 命中返回 → 未命中查 PG 并回填 Redis，设 TTL
- **写**：更新 PG → 删除对应 Redis key（下次读冷启动回填）
- **不预填**：不主动写缓存，避免缓存与 DB 不一致

### 4.3 限流策略

登录接口限流：Redis INCR 计数，同一 IP 5 次失败后锁定 15 分钟。滑动窗口，每个 key 独立过期。

```python
async def check_login_rate_limit(redis: Redis, ip: str):
    key = f"ratelimit:login:{ip}"
    attempts = await redis.incr(key)
    if attempts == 1:
        await redis.expire(key, 900)  # 首次计数设 15min 窗口
    if attempts > 5:
        raise Forbidden("登录尝试过于频繁，请 15 分钟后重试")
```

### 6.1 模块分解原则

- `router.py`：路由定义 + 参数校验（Pydantic schema），不写业务逻辑
- `service.py`：核心业务逻辑，接收 DB session + 参数，不感知 HTTP
- `schemas.py`：Pydantic 请求/响应模型
- `models.py`：SQLAlchemy ORM 模型（Table 定义），仅本模块拥有的表

跨模块引用的模型（如 consultation 的 Visit 被 billing 引用）通过 SQLAlchemy 的 `ForeignKey` 和 relationship 自然连接，不需要重复导入模型定义。`employee` 放在哪个模块？放在 `auth` 模块，因为 employee 和 account 是认证体系的起点。

## 七、认证与鉴权

### 7.1 登录流程

```
前端提交 username + password
        │
        ▼
POST /api/auth/login
        │
        ▼
auth/service.py
    1. 查 PG account 表 → password_hash（bcrypt）
    2. bcrypt.verify(password, password_hash) → 失败返回 401
    3. 联查 employee.role
    4. 签发 JWT: { sub: account_id, role, exp }
    5. UPDATE account.last_login = NOW()
    6. 返回 { access_token, user: { account_id, name, role } }
        │
        ▼
前端
    token 存 Zustand auth store + localStorage
    后续请求 Axios 拦截器自动注入 Authorization: Bearer <token>
    React Router 路由守卫读 role → 无权限跳 403
```

### 7.2 后端依赖注入

```python
# core/security.py — 两个核心依赖

async def get_current_user(
    token: str = Depends(oauth2_scheme),
    db: AsyncSession = Depends(get_pg_db),
    redis: Redis = Depends(get_redis)
) -> User:
    """解密 JWT，验黑名单，返回当前用户"""
    # 1. 解密 JWT，取出 account_id + role
    # 2. 查 Redis 黑名单 → 命中则 401
    # 3. 返回 User(account_id, role)
    ...

def require_role(*roles: str):
    """依赖注入工厂：只允许指定角色访问"""
    async def checker(user = Depends(get_current_user)):
        if user.role not in roles:
            raise Forbidden()
        return user
    return checker

# 使用
@router.get("/{id}")
async def get_customer(id: int, user = Depends(require_role("管理员", "医生", "护士"))):
    ...
```

### 7.3 角色权限矩阵

| 功能 | 管理员 | 医生 | 护士 |
|------|--------|------|------|

### 7.4 安全措施

| 环节 | 措施 |
|------|------|
| 密码存储 | bcrypt 12 轮哈希，永不存明文 |
| 登录防暴力 | Redis 记录每个 IP 失败次数，5 次锁定 15 分钟 |
| JWT 主动失效 | 修改密码/管理员禁用账号 → account_id 加入 Redis 黑名单，TTL = JWT 剩余有效时间 |
| 接口授权 | FastAPI 双层 dependency：`get_current_user` 验登录 → `require_role` 验角色 |
| 301/302 重定向 | 禁止，所有重定向手动处理，避免未授权访问 |

## 八、核心业务流程

### 8.1 就诊流程（挂号 → 诊断 → 处方 → 收费）

```
1. 挂号（管理员）
   POST /consultation/visits
   → INSERT visit (pet_id, employee_id=医生ID, status='待接诊')

2. 接诊（医生）
   PUT /consultation/visits/{id}/accept
   → UPDATE visit.status = '接诊中'
   前置条件：visit.status == '待接诊'

3. 诊断（医生）
   POST /consultation/visits/{id}/diagnosis
   → PG: INSERT diagnosis (visit_id, diagnosis_result, notes)
   → Mongo: INSERT medical_records (diagnosis_id, visit_id, sections)
   → PG: UPDATE visit.status = '待收费'
   前置条件：visit.status == '接诊中'

4. 开药（医生，可选）
   POST /consultation/diagnoses/{id}/prescriptions
   → PG: INSERT prescription_item (diagnosis_id, batch_id, quantity, dosage)
   → PG: SELECT ... FOR UPDATE medicine_batch, stock_qty -= quantity
   前置条件：库存足够（否则 409 + 事务回滚）
   如果有触发器依赖，已在设计中移除，库存扣减完全由后端事务控制

5. 生成收费项（医生）
   POST /billing/visits/{visit_id}/items
   → billing_service.ensure_bill(visit_id)
   → billing_service.add_item(visit_id, source_type, source_id, amount, description)
   幂等：UNIQUE(bill_id, source_type, source_id) 防重复

6. 结账（管理员）
   POST /billing/bills/{bill_id}/settle
   → PG: UPDATE bill.status = '已结清'
   → Celery: 生成账单 PDF → MinIO
   前置条件：bill.status == '未结清'
```

### 8.2 住院流程（住院 → 每日护理 → 出院）

```
1. 转入住院（医生/管理员）
   POST /hospitalization
   → PG: INSERT hospitalization (visit_id, ward_id, admit_date, status='住院中')
   → PG: UPDATE ward.status = '占用'
   前置条件：visit 已完成诊断，ward.status == '空闲'，同一 visit 不重复住院（UNIQUE 约束）

2. 日常护理（护士）
   POST /hospitalization/{hosp_id}/nursing
   → PG: INSERT nursing_record (hosp_id, employee_id, content)
   → Mongo: INSERT nursing_logs (record_id, hosp_id, content)
   前置条件：hosp.status == '住院中'

3. 出院（医生/管理员）
   PUT /hospitalization/{hosp_id}/discharge
   → PG: UPDATE hospitalization (discharge_date=NOW(), status='已出院')
   → PG: UPDATE ward.status = '空闲'
   → billing_service.ensure_bill(visit_id)
   → billing_service.add_item(visit_id, 'hospitalization', hosp_id, amount, f"住院费 {days}天")
      其中 amount = ward.daily_rate × (discharge_date - admit_date)
   前置条件：hosp.status == '住院中'
```

### 8.3 药品入库流程

```
POST /pharmacy/batches
   → PG: INSERT medicine_batch (medicine_id, in_date, expire_date, stock_qty, cost_price)
   前置条件：expire_date > in_date, stock_qty >= 0, cost_price >= 0

库存扣减：
   仅由处方开具流程（8.1 步骤 4）触发，使用 SELECT ... FOR UPDATE 行级锁
   库存不足时事务回滚，返回 409 "库存不足，当前库存 N"
```

### 8.4 状态流转规则汇总

**visit.status:**
```
待接诊 ──(accept)──→ 接诊中 ──(diagnosis)──→ 待收费 ──(settle)──→ 已完成
   │                    │
   └──(cancel)──→ 已取消  └──(cancel, 无诊断)──→ 已取消
```

**hospitalization.status:**
```
住院中 ──(discharge)──→ 已出院
```

**bill.status:**
```
未结清 ──(settle)──→ 已结清
```

## 九、API 设计规范

### 9.1 统一响应格式

```json
// 成功
{ "code": 200, "data": { ... }, "message": "ok" }

// 业务错误
{ "code": 409, "data": null, "message": "库存不足，当前库存 3" }

// 校验错误
{ "code": 422, "data": [{ "field": "phone", "msg": "格式无效" }], "message": "参数校验失败" }

// 权限拒绝
{ "code": 403, "data": null, "message": "无权限" }

// 未找到
{ "code": 404, "data": null, "message": "就诊记录不存在：42" }
```

### 9.2 异常体系

```python
# core/exceptions.py

class AppError(Exception):
    def __init__(self, code: int, message: str):
        self.code = code
        self.message = message

class NotFound(AppError):
    def __init__(self, entity: str, id: int):
        super().__init__(404, f"{entity}不存在：{id}")

class Conflict(AppError):
    def __init__(self, message: str):
        super().__init__(409, message)

class Forbidden(AppError):
    def __init__(self):
        super().__init__(403, "无权限")

@app.exception_handler(AppError)
async def app_error_handler(request, exc):
    return JSONResponse(
        status_code=exc.code,
        content={"code": exc.code, "data": None, "message": exc.message}
    )
```

### 9.3 前端错误处理

| HTTP code | 前端行为 |
|-----------|---------|
| 400 | 表单校验失败，字段高亮错误提示 |
| 401 | 清空 token → 跳转 `/login` |
| 403 | toast "无权限访问" |
| 404 | toast 具体实体名称 |
| 409 | toast `message`（库存不足、状态不可操作等） |
| 422 | 表单字段绑定后端校验错误 |
| 5xx | toast "服务异常，请稍后重试" |

Axios 响应拦截器统一处理，对用户始终有 toast 反馈。

### 12.3 分支策略

按领域模块切 feature branch，一个模块做完合并回 main：

```
main
  feat/infra        → main
  feat/auth         → main
  feat/customer     → main
  feat/consultation → main
  feat/billing      → main
  feat/hospitalization → main
  feat/pharmacy     → main
  feat/boarding     → main
```

每个分支聚焦一个模块，独立开发、测试后合并。不做长周期分支。

### 12.4 推荐实现顺序

```
1. infra        core/config, pg_db, mongo_db, redis, minio, deps, security, exceptions
2. auth         登录/登出/JWT中间件/角色守卫
3. customer     客户 + 宠物 CRUD（最独立，零依赖）
4. consultation 挂号 → 诊断 → 处方（核心链，依赖 customer + auth）
5. billing      结账 + 收费项（依赖 consultation）
6. hospitalization  住院 + 护理（依赖 consultation + billing）
7. pharmacy     药品 + 入库 + 库存扣减（独立模块，但处方依赖扣减逻辑）
8. boarding     寄养（独立模块，依赖 customer + ward）
9. export       账单 PDF 导出 + Celery 异步任务
```

`auth → customer → consultation → billing` 是硬依赖链，前面不跑通后面没法测。`pharmacy` 的扣减代码在 consultation 模块（处方）调用，但 pharmacy 自身可先开发。

## 十三、已知设计决策与权衡

| 决策 | 选择 | 权衡 |
|------|------|------|
| 不拆微服务 | 领域模块化单体 | 降低初期复杂度，目录边界清晰，未来可平滑拆分 |
| MongoDB 存文书 | 1:1 映射 PG record | 避免跨库外键，PG 可独立运行 |
| Cache-Aside 而非 Write-Through | 写时删缓存 | 实现简单，避免写不一致，miss 惩罚可接受 |
| PG 触发器移除 | 库存扣减归后端 | 业务逻辑集中，可测试可调试，`CHECK` 保留做最后防线 |
| bill_item 幂等 | source_type + source_id 唯一 | 防御前端重试/网络抖动，DB 层保证 |
| visit 与 hospitalization 状态分离 | hospitalization 独立 status，visit 不冗余存储 | 住院状态由 hospitalization 权威维护，查询 EXISTS 即可，无数据冗余 |
| 取消规则 | 有诊断就不能取消 | 保护医疗数据完整性 |
